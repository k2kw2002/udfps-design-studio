## 1. PINN 모델 구조

### Q: PINN의 Input/Output은?

**Input:** BM 내부의 위치 좌표 (x, z)
- x = 센서 면 가로 위치
- z = 두께 방향 위치 (0~30μm)

**Output:** 그 위치에서의 전자기장 U(x,z)
- U_r = 빛의 세기 (실수부)
- U_i = 빛의 방향 (허수부)
- 센서 위치(z=30μm)에서의 출력이 곧 **PSF**

```
Input              SIREN 신경망              Output
(x, z)  →  [4층 × 128유닛, sin 활성화]  →  (U_r, U_i)
좌표                                        전자기장

→ z=30μm에서의 |U_r² + U_i²| = PSF (최종 출력)
```

### Q: 사용한 프레임워크는?

**순수 PyTorch로 직접 구현** (DeepXDE, NVIDIA Modulus 미사용)

| 프레임워크 | 사용 여부 | 이유 |
|---|---|---|
| PyTorch (순수) | ✅ 사용 | L_phase★ 커스텀 Loss 구현 자유도 |
| DeepXDE | ❌ | 범용 PINN 라이브러리, 커스텀 어려움 |
| NVIDIA Modulus | ❌ | GPU 서버 필요, 현재 환경에 부적합 |

Phase 2에서 A100 GPU 도입 시 NVIDIA Modulus 전환 검토.

## 2. 물리 파이프라인 흐름

### Q: 전체 계산 흐름은?

```
[TMM]  AR 코팅 위상 Δφ 계산 (PINN 밖에서 미리 계산)
  ↓
[ASM]  Cover Glass 550μm 전파 (PINN 밖에서 미리 계산)
  ↓
  U_in(x) = 빛이 BM 입구에 도착한 상태
  ↓
[PINN] BM 내부 30μm 영역에서 빛 분포 예측 (PINN이 학습)
  ↓
  U_out(x) = 센서에 도달한 빛 = PSF
```

PINN은 **BM 내부(z=0~30μm)만** 담당한다. TMM과 ASM이 BM 입구까지 빛을 가져다주고, PINN이 BM 내부의 회절을 계산한다.

## 3. Loss 함수 상세

### Q: L_phase는 어디서 비교하는가?

**z=0 (BM 입구)에서만 비교한다.**

```
z=0 (BM 입구): TMM이 계산한 위상과 PINN 예측 위상을 비교 → L_phase
z=0~30 (내부): 물리법칙(Helmholtz)으로 스스로 학습 → L_pde
z=30 (출구):   PINN이 예측한 결과 → PSF (최종 출력)
```

| Loss | 비교 위치 | 비교 내용 |
|---|---|---|
| L_phase★ | z=0 (입구) | PINN 위상 vs TMM 위상 |
| L_data | z=0 (입구) | PINN 밝기 vs LightTools/ASM 밝기 |
| L_pde | 내부 전체 | Helmholtz 방정식 만족 여부 |
| L_bm | BM 벽 위치 | U=0 (빛 차단) 여부 |

### Q: PINN 출력과 LightTools 출력이 다른데 비교 가능한가?

가능하다. PINN 출력에서 밝기를 꺼내 비교한다.

```
PINN 출력:       U_r, U_i (실수부, 허수부)
                     ↓
                 U_r² + U_i² = 밝기 (강도)
                     ↓
LightTools 출력: I(x,y) = 밝기
                 → 이 둘을 비교 = L_data
```

PINN이 밝기와 방향을 **동시에** 갖고 있기 때문에:
- 밝기 → LightTools와 비교 (L_data)
- 방향 → TMM과 비교 (L_phase★)

이것이 PINN의 핵심 강점: LightTools(밝기만)와 TMM(방향만)을 **하나로 통합**.

## 4. BM 오프셋 원리

### Q: BM1을 왜 왼쪽(-20μm)으로 옮기는가?

θ=30°로 비스듬히 입사한 빛은 **오른쪽으로 치우쳐서** 센서에 도달한다.

```
θ=+30° 빛:
    ╲ 비스듬한 빛
     ╲
      → 오른쪽에 빛이 더 많음 = PSF 비대칭

해결: BM1 구멍을 왼쪽으로 이동
      → 오른쪽 치우친 빛을 잘라냄
      → 통과하는 빛이 대칭에 가까워짐
```

빛을 옮기는 것이 아니라, **구멍을 옮겨서 치우친 부분을 잘라내는** 원리이다.

### Q: 빛을 자르면 광량이 줄지 않나?

맞다. 그래서 **3목적 동시 최적화**가 필요하다.

```
delta_BM을 많이 옮기면:
  skewness ↓ (좋아짐)  BUT  광량 ↓ (나빠짐)
  → Trade-off 관계
```

이를 코팅 최적화로 보상한다:
- BM 오프셋: 빛 잘라서 대칭으로 → 광량 약간 감소
- 코팅 최적화: 투과율 +57% 증가 → 광량 보충

→ 결과: 대칭이면서 밝기도 유지

### Q: θ=-30°면 반대로 치우치지 않나?

맞다. 이것이 Phase 2에서 해결해야 할 핵심 문제이다.

```
θ=+30° → 오른쪽 치우침 → delta=-20 필요
θ=-30° → 왼쪽 치우침  → delta=+20 필요

하나의 delta로 양쪽을 동시에 보정할 수 없음!
```

Phase 2 해결 방향:
1. **Robust Design**: 모든 각도에서 평균적으로 최선인 delta 탐색
2. **비대칭 BM 구조**: 아퍼처 모양 자체를 비대칭으로 설계
3. **픽셀별 BM 오프셋**: 센서 위치에 따라 다른 delta 적용

## 5. PSF 비대칭 원인 분리

### Q: 코팅을 바꾸면 PSF가 대칭이 되나?

아니다. 코팅 변경은 **광량(T_total)에만 영향**을 주고, PSF 비대칭은 변하지 않는다.

| 요인 | 코팅 두께 변경 효과 | BM 구조 효과 |
|---|---|---|
| T_total (광량) | +57% 개선 | 영향 없음 |
| skewness (비대칭) | 거의 영향 없음 | **지배적 원인** |

→ PSF 비대칭의 주 원인은 **BM 구조(회절)**이다.
→ 코팅 최적화(광량 확보)와 BM 오프셋(비대칭 해소)을 분리하여 각각 최적화하는 전략 수립.

## 6. 1D vs 2D 확장

### Q: "1D → 2D 확장"이란?

**PSF 출력 기준**으로 말하는 것이다. (센서 면에서 본 차원)

| | PINN 계산 공간 | PSF 출력 | 표현 |
|---|---|---|---|
| 현재 | (x, z) = 2D 단면 | PSF(x) = 1D | "1D PSF" |
| Phase 2 | (x, y, z) = 3D 공간 | PSF(x,y) = 2D | "2D PSF" |

위에서 내려다본 센서 면:
- **현재**: x 방향 한 줄만 계산
- **Phase 2**: x + y 면 전체 계산

```
현재 (1D PSF):        Phase 2 (2D PSF):
                      y
x 방향만 계산          │  ○○○○○
━━━━━━━━━━           │  ○●●●○    x + y 방향
                      │  ○●●●○    모두 계산
                      └──────→ x
```

### Q: 왜 A100 GPU가 필요한가?

계산 포인트가 800배 증가한다.

```
현재:    x × z     = 800 × 100       = 80,000 포인트
Phase 2: x × y × z = 800 × 800 × 100 = 64,000,000 포인트
```
